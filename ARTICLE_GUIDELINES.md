# ARTICLE_GUIDELINES — Claude Academy

מסמך זה מגדיר את הסטנדרט לכתיבת תוכן חדש באתר academy.dermunbound.com. כל יחידת תוכן חייבת לעבור את ה-AUDIT שבסוף המסמך לפני merge ו-build.

---

## 1. קהל יעד

**דמות הקורא (persona):** רופא/ה עברי-דובר או דו-לשוני, עובד/ת במרפאה פרטית או בית חולים בישראל. בעל/ת היכרות בסיסית עם ChatGPT אבל לא עם Claude. בעל/ת הבנה טכנולוגית של "משתמש מתקדם" — יודע/ת להתקין אפליקציה, להעלות PDF, להשתמש בדפדפן. **לא קורא/ת קוד, לא מכיר/ה terminal.**

**מה זה אומר לפרקטיקה:**

- מונחים טכניים — תמיד עם הסבר חד-משפטי בגוף הטקסט, לא רק ב-`:::advanced`.
- קוד — רק כשאי אפשר בלי (פרומפט או פקודה בסיסית). לעולם לא סקריפטים ארוכים.
- כל שיעור פותח בשאלה רפואית, לא בהסבר טכני.
- ברירת המחדל של גוף הטקסט היא **"רופא-לא-מפתח"**. השתמש ב-`:::beginner` לחיזוק עדין, ו-`:::advanced` להרחבה אופציונלית.

## 3. כללי כתיבה

### 3.1 פתיחה

כל יחידת תוכן פותחת בשאלה רפואית. לא ב"סקירה כללית של X", לא ב"בשיעור הזה נלמד".

**טוב:**
> "האם יצא לך לחזור הביתה בסוף יום מרפאה עם 12 דוחות פתולוגיה לסרוק? בשיעור הזה נראה איך Claude Code יכול לעשות את זה בשבילך ב-3 דקות."

**רע:**
> "בשיעור הזה נסקור את היכולות של Claude Code ואת היתרונות שלו."

### 3.2 מבנה הגוף

1. **בעיה רפואית קונקרטית** (שאלת פתיחה)
2. **מה הכלי עושה** (2-3 משפטים, בשפה פשוטה)
3. **Use case #1 עם פרומפט מלא לדוגמה** (תוך 3 פסקאות מהפתיחה!)
4. **2-4 use cases נוספים** — בקצרה
5. **הגבלות ואזהרות** (פרטיות, דיוק, HIPAA/חוק הגנת הפרטיות)
6. **מה לקרוא/לצפות הלאה**

### 3.3 רמת קורא — מתי להשתמש ב-callouts

- **גוף הטקסט** — רמת "רופא-לא-מפתח". מונחים טכניים עם הסבר חד-משפטי מיד אחריהם.
- **`:::beginner`** — חיזוק למי שחדש לגמרי ב-AI (דוגמה: "אם זו הפעם הראשונה שלך עם AI, חשוב על זה ככלי עזר שהתאמן על טונות של טקסט, ולכן הוא יודע להשיב לשאלות בשפה טבעית.")
- **`:::advanced`** — העמקה טכנית אופציונלית (דוגמה: "טכנית, MCP עובד על JSON-RPC מעל stdio או HTTP+SSE, ואפשר לכתוב servers ב-Python או TypeScript.")

**כלל אגודל:** אם הסרת את ה-callouts וגוף הטקסט עדיין מובן לרופא-לא-מפתח — המבנה תקין.

### 3.4 Use cases רפואיים — הלב של כל יחידה

כל יחידת תוכן חייבת **לפחות 3 use cases רפואיים ספציפיים**. לא "רופאים יכולים להשתמש בזה ל..." אלא תרחיש קונקרטי:

**טוב:**
> "מטופלת בת 62, דרמטומיוזיטיס ידועה, מגיעה עם קוצר נשימה מתמשך. העלה את רשימת התרופות שלה ואת תוצאות ה-HRCT ל-Claude Desktop ובקש: 'האם התמונה מתאימה ל-ILD משני ל-DM? מה הבירור הבא לפי הגיידליינים של EULAR 2024?'"

**רע:**
> "רופאים יכולים להשתמש ב-Claude לאבחנה מבדלת."

### 3.5 מקורות

כל טענה קלינית חייבת מקור מהימן. מקורות מוכרים:

- **PubMed** — כל מאמר עם PMID
- **UpToDate, DynaMed, BMJ Best Practice** — מותר לציין גם אם אין קישור חופשי
- **guidelines של אגודות מקצועיות:** AAD, EADV, NCCN, EULAR, משרד הבריאות הישראלי
- **docs רשמיים של Anthropic:** docs.claude.com, support.anthropic.com, code.claude.com

**פורמט מקורות בסוף יחידה:**

```markdown
## מקורות

1. Author et al. Title. *Journal*. Year;Volume:Pages. https://pubmed.ncbi.nlm.nih.gov/PMID/
2. Anthropic. Claude Desktop documentation. https://docs.claude.com/...
3. משרד הבריאות. חוזר מנכ"ל ... https://www.gov.il/...
```

**טענות שאסור לפרסם בלי מקור:**
- מינוני תרופות
- קווים מנחים ("לפי הגיידליינים של X")
- סטטיסטיקות ("30% מהמטופלים...")
- התוויות / התוויות נגד
- טענות יעילות ("יעיל ב-X% מהמקרים")

## 4. כללי RTL ועברית

אלו הכללים שהכי הרבה פעמים נשברים. קרא בעיון.

### 4.1 לא להתחיל משפט עברי באנגלית

**רע:**
> "Claude Desktop הוא האפליקציה החדשה של Anthropic."

**טוב:**
> "האפליקציה החדשה של Anthropic, Claude Desktop, משנה את הדרך שבה רופא..."

**גם טוב:**
> "‫Claude Desktop‬ — האפליקציה החדשה של Anthropic — משנה את הדרך..."
(עטיפת ה-Latin במעטפת `‫...‬` — אבל רק במצבים מיוחדים.)

### 4.2 לא לכתוב מספרים עירומים באמצע משפט עברי

**רע:**
> "הגרסה החדשה תומכת ב-1000000 טוקנים."

**טוב:**
> "הגרסה החדשה תומכת בחלון הקשר של מיליון טוקנים."

**גם טוב:**
> "הגרסה החדשה תומכת בחלון הקשר של כ-1M טוקנים (מיליון)."

**אם חייבים מספר עירום, השתמש בתו NBSP (`\u00A0`) לפני ואחרי:**
> `תומכת ב-1,000,000\u00A0טוקנים (מיליון).`

### 4.3 שמות מוצרים באנגלית

שמות מוצרים נשארים באנגלית. **אל תתרגם:**
- Claude, Claude Code, Claude Desktop, Claude Pro, Claude Max
- Cowork, Dispatch, Routines
- MCP, Anthropic
- ChatGPT, Gemini, Copilot
- שמות שפות תכנות: Python, JavaScript

**הוסף glossary קצר בפעם הראשונה:**
> "Claude (קלוד) הוא עוזר מבוסס בינה מלאכותית שפותח על ידי Anthropic."

בהופעות הבאות — רק `Claude`.

### 4.4 תעתיק עברי — מתי כן ומתי לא

**כן לתעתיק:**
- בכותרות ראשיות (לחוויה קוראית) — "קלוד במרפאה"
- בתוך משפט עברי זורם שבו האנגלית מפריעה

**לא לתעתיק (להשאיר Latin):**
- שמות מוצרים ראשוניים בהגדרה (ראה 4.3)
- פקודות, פרומפטים, קוד
- שמות קבצים, נתיבים, URLs

**רשימת תעתיקים מוסכמים** (השתמש בהם באופן אחיד):

| Latin | עברית מותרת (לא-הגדרתית) | הערה |
|---|---|---|
| Claude | קלוד | ראשית ההופעה — Claude (קלוד). הבאות — Claude. |
| Claude Code | קלוד קוד | רק בכותרות. בגוף — Claude Code. |
| Claude Desktop | קלוד דסקטופ | רק בכותרות. בגוף — Claude Desktop. |
| Cowork | קואורק / Cowork | עדיפות ל-Cowork. |
| Dispatch | דיספטצ' / Dispatch | עדיפות ל-Dispatch. |
| prompt | פרומפט | מותר בגוף. |
| token | טוקן | מותר בגוף. |
| context window | חלון הקשר | תרגום מלא, לא תעתיק. |
| hallucination | הזיה | תרגום מלא. |
| agent | סוכן | תרגום מלא. |

### 4.5 כיוון בתוך רכיבים מעורבים

אם שיעור/פוסט מכיל רצפי אנגלית ארוכים (פרומפט, פקודה), עטוף אותם בבלוק קוד (\`\`\`). זה פותר את ה-RTL אוטומטית כי הרכיב כבר מוגדר `dir="ltr"`.

**רע (ריצת אנגלית ב-inline):**
> "הפקודה שלנו היא npm install -g @anthropic-ai/claude-code --verbose --global, וזה יתקין הכל."

**טוב (בלוק קוד):**
> "הפקודה שלנו היא:
> ```
> npm install -g @anthropic-ai/claude-code --verbose --global
> ```
> וזה יתקין הכל."

### 4.6 סימני פיסוק

- **אין em-dashes (`—`) בעברית.** השתמש ב-hyphen רגיל (`-`) עם רווחים: "Claude - הכלי החדש".
- **אין מירכאות לועזיות ("...").** השתמש במירכאות עבריות ("...") או גרש בודד ('...').
- **נקודה בסוף כל משפט** (גם ברשימת bullets).

## 5. שפה ופשטות — בלי סימני AI

### 5.1 מילים וביטויים אסורים (AI markers)

**באנגלית:**
- ❌ "comprehensive" (אלא אם שם פרטי)
- ❌ "delve into"
- ❌ "leverage"
- ❌ "paradigm"
- ❌ "seamlessly" / "seamless"
- ❌ "robust" (בתור adjective כללי)
- ❌ "cutting-edge"
- ❌ "game-changing" / "game-changer"
- ❌ "in today's fast-paced world"
- ❌ "navigate the complexities"

**בעברית:**
- ❌ "מקיף ו..." / "בצורה מקיפה"
- ❌ "בסופו של דבר" (בתור פתיחת פסקה)
- ❌ "למעשה" (כ-filler)
- ❌ "ראוי לציין ש..."
- ❌ "בעידן הנוכחי"
- ❌ "מרתק" / "מעניין במיוחד"
- ❌ "חשוב לזכור ש..." (אלא אם זו אזהרה אמיתית)

### 5.2 ניסוחים עדיפים

- במקום "Claude Desktop מציע מגוון רחב של יכולות" → "Claude Desktop עושה X, Y ו-Z."
- במקום "יכול לשפר באופן משמעותי את..." → "חוסך לך X דקות ביום."
- במקום "מהפכני" → "חדש" (או שם הפיצ'ר).

### 5.3 כלל "המשפט הקצר"

אם משפט בעברית עובר שתי שורות, פרק אותו. רופא עייף שקורא ב-13:45 בין מטופלים לא ינצח משפט של 40 מילים.

## 6. דיוק רפואי

### 6.1 לא להמציא נתונים

אם אתה לא בטוח במספר, סטטיסטיקה, או אמירה קלינית — או תחפש מקור ותצטט אותו, או תשכתב בלי המספר. אל תמציא ("בערך 30% מהמקרים").

### 6.2 לא להחליף רופא

בכל מקום בו מוצגת אינטראקציה קלינית, חייבת להופיע הבהרה:

> "Claude הוא כלי עזר ואינו מחליף את השיפוט הקליני שלך. אמת כל המלצה קלינית מול מקור מהימן (UpToDate, קווים מנחים עדכניים, או התייעצות עם עמית) לפני החלטה לגבי מטופל."

ניסוח זה חייב להופיע בסוף כל יחידת תוכן רפואית-קלינית.

### 6.3 פרטיות מטופלים (חוק הגנת הפרטיות תשמ"א-1981, תיקון 13)

כל יחידה שמציגה שימוש קליני חייבת לכלול סקציה "**פרטיות והגנה על נתוני מטופלים**" עם:

- אזהרה לא להעלות PII (שם, ת.ז., טלפון, תאריך לידה מדויק) ל-Claude אלא אם המסלול מאובטח.
- הבהרה ש-Claude.ai הוא מסלול לא-מאובטח לנתונים רפואיים בלי BAA/DPA.
- המלצה ל-deidentification לפני העלאה.
- הפניה לפרוטוקול ה-privacy של Anthropic ולחוק הישראלי.

### 6.4 זמינות תרופות בישראל

לפני שאתה מזכיר תרופה בשמה המסחרי — ודא שהיא זמינה בישראל (מרשם/סל בריאות). אם אתה לא בטוח — ציין את החומר הפעיל בלבד ("clobetasol propionate 0.05%") ולא שם מסחרי.

## 7. דו-לשוניות — עברית ואנגלית

### 7.1 שוויון

כל יחידה חייבת להופיע בשתי השפות. עברית ואנגלית **באותה רמת איכות ועומק** — לא תרגום-מכונה, לא תקציר חלקי.

### 7.2 ההבדלים המותרים

- **דוגמאות רפואיות** — מותר להחליף דוגמה ישראלית ב-EN (למשל: "דוח פתולוגיה מ-Maccabi" בעברית → "Pathology report from a hospital system" באנגלית).
- **רשויות מקומיות** — "משרד הבריאות" בעברית → "Israel Ministry of Health" באנגלית.
- **טון** — האנגלית יכולה להיות יותר מקצועית-נקייה, העברית יכולה להיות קצת יותר חמה.

### 7.3 מה אסור להשתנות

- רשימת features
- מספר use cases
- מקורות (אותם מקורות בשתי השפות)
- הנחיות פרטיות ואזהרות קליניות
- מבנה החלוקה לפסקאות

## 8. AUDIT checklist — רשימת בדיקה לפני פרסום

בסיום כתיבה, עבור על הרשימה הזו **פעמיים**: פעם אחת מיד אחרי כתיבה, ופעם שנייה לפני commit.

### 8.1 Audit ראשוני (מיד אחרי כתיבה)

- [ ] פתיחה בשאלה רפואית קונקרטית
- [ ] גוף הטקסט מובן לרופא-לא-מפתח גם בלי ה-callouts
- [ ] לפחות 3 use cases רפואיים ספציפיים
- [ ] פרומפט דוגמה מלא ב-code block, לא inline
- [ ] סקציית "מקורות" בסוף עם לפחות 3 מקורות
- [ ] disclaimer "Claude הוא כלי עזר..." בסוף יחידה קלינית
- [ ] סקציית "פרטיות והגנה על נתוני מטופלים" אם יש שימוש קליני
- [ ] excerpt נכתב בדיוק (1-2 משפטים, לא יותר)
- [ ] readTime מדויק (חשב ~200 מילים לדקה)

### 8.2 Audit סגנוני (Anti-AI markers)

חפש במסמך (Ctrl+F) כל אחד מהבאים ותקן:

- [ ] `comprehensive` / "מקיף"
- [ ] `delve` / "להעמיק"
- [ ] `leverage` / "למנף"
- [ ] `seamless` / "חלק"
- [ ] `robust`
- [ ] em-dashes (`—`)
- [ ] "בסופו של דבר"
- [ ] "ראוי לציין"
- [ ] "למעשה" (כ-filler)

### 8.3 Audit RTL ועברית

- [ ] אף משפט עברי לא מתחיל ב-Latin
- [ ] אף מספר עירום באמצע משפט עברי (ראה 4.2)
- [ ] תעתיקים עקביים (קלוד / Claude — בחר כלל אחד ליחידה)
- [ ] רצפי אנגלית ארוכים רק בבלוקי קוד
- [ ] מירכאות עבריות ("..."), לא לועזיות ("...")
- [ ] נבדק במסך mobile (או preview) ב-RTL

### 8.4 Audit תרגום (HE ↔ EN)

- [ ] מספר פסקאות זהה
- [ ] מספר use cases זהה
- [ ] אותם מקורות בסוף
- [ ] אותן אזהרות/disclaimers
- [ ] כותרות H2/H3 מקבילות

### 8.5 Audit רפואי

- [ ] כל טענה קלינית עם מקור מוכר
- [ ] אין מינוני תרופות מומצאים
- [ ] שמות תרופות — חומר פעיל (לא שם מסחרי), או מאומתות לשוק הישראלי
- [ ] לא הוצגה אינטראקציה שמחליפה שיפוט קליני
- [ ] disclaimer הוצג היכן שנדרש

### 8.6 Audit טכני (Anthropic features)

## 9. תהליך כתיבה מומלץ

1. **קבעו use cases לפני שהתחלתם לכתוב** — 3-5 תרחישים רפואיים קונקרטיים.
2. **מצאו מקורות לכל טענה קלינית** — לפני, לא אחרי.
3. **ודאו feature validity** — אם הפוסט על Claude Desktop/Cowork/Code, חפשו את ה-docs הרשמיים עכשיו. שמרו לינק.
4. **כתבו טיוטה ב-EN** (לרוב קל יותר), אחר כך תרגמו/התאימו ל-HE.
5. **עברו על AUDIT 8.1** — תיקונים.
6. **שמרו וסגרו את הקובץ**. קחו הפסקה של 15 דקות.
7. **חזרו, עברו על AUDIT 8.2-8.6** — תיקונים.
8. **הריצו `npm run build`** — תיקון שגיאות אם יש.
9. **בדיקה ויזואלית** — `npm run dev` וקראו את היחידה במסך, גם mobile.

## 10. מה לעשות כשתוקעים

- **אם אתה לא בטוח בטענה קלינית** → פתח את UpToDate/PubMed. אם אתה עדיין לא בטוח, השמט את הטענה.
- **אם RTL נשבר לך** → הוסף בלוק קוד סביב ה-Latin הארוך, או שכתב את המשפט כך שיתחיל בעברית.
- **אם סיום המאמר נשמע AI-ish** → מחק את פסקת הסיכום. סיים עם המקורות. רוב הפוסטים לא צריכים סיכום.
- **אם אתה לא יודע איך לתרגם מונח ל-HE** → השתמש בטבלת התעתיקים (4.4). אם המונח לא שם, השאר באנגלית והוסף glossary בהופעה הראשונה.
